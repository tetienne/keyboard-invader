# Phase 8.2: Word Mode UX Polish - Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Children see per-character visual feedback while typing words in word mode, and the menu highlights their preferred game mode. This is a UX polish pass on existing word mode gameplay and menu UI — no new game mechanics, no new screens.

</domain>

<decisions>
## Implementation Decisions

### Per-Character Typing Feedback
- **D-01:** Each correctly typed character changes from white to **gold/yellow (#FFD700)** tint. Untyped characters remain white. The color split between gold and white serves as the cursor — no additional cursor indicator needed.
- **D-02:** No animation per character — pure color change on correct key. Keeps it simple and performant.
- **D-03:** Implementation requires switching from a single `BitmapText` to **SplitBitmapText** (or equivalent per-character tint approach) since standard BitmapText applies tint uniformly to all characters. The current `wordBt` in `_spawnWord()` (`states.ts:750-757`) needs refactoring.
- **D-04:** The `cursorIndex` on `WordEntity` already tracks progress — use it to determine which characters are gold vs white on each frame or on each correct keypress.

### Menu Preferred Mode Highlight
- **D-05:** The preferred mode button gets a **subtle animated glow** (pulsing border/aura) around its space panel. Consistent with the sci-fi glow style established in Phase 8 (D-21).
- **D-06:** The non-preferred button stays in its normal state — no dimming or de-emphasis.
- **D-07:** On first launch when `preferredGameMode` is `null`: **no highlight on either button**. Both buttons look equal. The glow appears only after the child has played at least one session.
- **D-08:** `preferredGameMode` is already persisted on `ProfileData` and set via `ctx.setGameMode()` in `profile-state.ts:251`. The `MenuState` just needs to read `ctx.getActiveProfile()?.preferredGameMode` and apply the glow accordingly.

### Claude's Discretion
- Exact glow animation parameters (pulse speed, opacity range, color)
- Whether to use PixiJS SplitBitmapText or a manual multi-BitmapText approach for per-character tinting
- Glow implementation technique (Graphics outline, filter, or tween-based alpha on a border graphic)
- Any minor code reorganization needed to support the changes

</decisions>

## Existing Code Insights

### Reusable Assets
- **`WordEntity.cursorIndex`** (`src/game/words.ts:35`): Already tracks typing progress per word — use directly for gold/white split
- **`drawSpacePanel()`**: Draws rounded space-themed panel for buttons — extend or layer a glow effect on top
- **`AlienContainer.wordLabel`** (`src/game/alien-container.ts`): BitmapText reference on each word alien — needs replacement with per-char approach
- **Tween system** (`src/game/tween.ts`): Can drive glow pulse animation on menu button

### Established Patterns
- **BitmapText for all text**: GameFont installed in BootState. Per-char tinting requires SplitBitmapText or manual character splitting
- **Space panel UI**: Menu buttons use `drawSpacePanel()` with `pointerover` scale feedback
- **Profile-aware context**: `ctx.getActiveProfile()` available in all states for reading preferences

### Integration Points
- `src/game/states.ts` `_spawnWord()` — Refactor word text from single BitmapText to per-char tintable approach
- `src/game/states.ts` `_processWordInput()` — After `cursorIndex++` on correct key, update character tints
- `src/game/states.ts` `MenuState.enter()` — Read preferred mode, apply glow to matching button
- `src/game/words.ts` — May need to update `WordEntity` interface if text representation changes

<specifics>
## Specific Ideas

- Gold/yellow (#FFD700) was chosen for typed characters because it feels like "progress" and XP, distinct from the green used for hit feedback in letter mode
- No cursor indicator — the gold/white boundary IS the cursor. Keeps the word clean and uncluttered for children
- Glow border on preferred mode follows the sci-fi control panel aesthetic from Phase 8
- First launch shows equal buttons — no assumptions about which mode the child prefers

</specifics>
