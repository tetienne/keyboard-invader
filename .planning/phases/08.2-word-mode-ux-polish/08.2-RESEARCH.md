# Phase 8.2: Word Mode UX Polish - Research

**Researched:** 2026-04-12
**Domain:** PixiJS per-character text tinting, menu UI glow effects
**Confidence:** HIGH

## Summary

This phase addresses two focused UX improvements: (1) per-character visual feedback in word mode using gold/white tinting, and (2) a glow highlight on the preferred game mode button in the menu. Both are well-scoped changes to existing code with clear integration points.

The critical technical finding is that `SplitBitmapText` -- the ideal PixiJS v8 API for per-character tinting -- is available in the installed PixiJS 8.17.1 and exports individual `BitmapText` children via its `.chars` array, each with an independent `.tint` property. The Phase 8 crash that led to its removal was caused by pool preallocation (creating PixiJS objects without a render context), not by SplitBitmapText itself. Since word aliens use lazy creation at spawn time (when already parented to the stage), SplitBitmapText should work correctly now.

The menu glow effect is straightforward: draw a second `Graphics` roundRect behind the preferred mode's space panel with the existing `SPACE_PALETTE.glow` color and animate its alpha using PixiJS ticker (already available in `MenuState.update()`).

**Primary recommendation:** Use `SplitBitmapText` for per-character word tinting (gold #FFD700 on typed chars), with a manual multi-BitmapText fallback approach documented in case of runtime issues. Use Graphics-based glow aura with ticker-driven alpha pulse for the menu highlight.

## Architecture Patterns

### SplitBitmapText Per-Character Tinting

**What:** Replace the single `BitmapText` word label with `SplitBitmapText` to get individual character tinting via the `.chars` array. Each entry in `.chars` is a `BitmapText` instance with its own `.tint` property. [VERIFIED: node_modules/pixi.js/lib/scene/text-bitmap/utils/bitmapTextSplit.js line 44 -- chars are BitmapText instances]

**Structure:** `SplitBitmapText` extends `Container`. It auto-splits text into a hierarchy:
- `lines[]` -- Container per line
- `words[]` -- Container per word (within a line)
- `chars[]` -- BitmapText per character (flat array, all chars across all words)

For single-word text (our use case), `chars.length === text.length` (excluding spaces). [VERIFIED: bitmapTextSplit.js source code]

**Caveat:** "When splitting characters, browser kerning is lost, so spacing may differ slightly compared to standard text." [CITED: https://pixijs.com/blog/8.11.0] This is acceptable for game word display.

**When to use:** When you need per-character visual properties (tint, alpha, scale, rotation) on BitmapText.

### Phase 8 SplitBitmapText Crash -- Root Cause Analysis

**What happened:** In Phase 8 (08-04-SUMMARY.md), SplitBitmapText crashed with a "parent chain error." This was listed alongside pool preallocation crashes. [VERIFIED: .planning/phases/08-visual-identity/08-04-SUMMARY.md:27,31]

**Root cause:** Pool preallocation created PixiJS objects before they had a parent chain / render context. The crash was about premature instantiation, not a fundamental SplitBitmapText bug.

**Why it is safe now:** Word pool items use lazy creation -- `_spawnWord()` only creates text when the AlienContainer is already added to the stage (`wordParent.addChild(ac)` at line 779, BitmapText creation at lines 750-757 happens inside `if (!ac.wordLabel)` the first time the item is spawned). The parent chain is established by the time text is created. [VERIFIED: src/game/states.ts:748-779]

**Risk mitigation:** If SplitBitmapText still crashes at runtime due to font resolution timing, fall back to manual approach (multiple individual BitmapText objects positioned manually). The mock already exists (`MockSplitBitmapText` in tests/__mocks__/pixi.ts:79). [VERIFIED: tests/__mocks__/pixi.ts:79-97]

### Glow Animation Pattern

**What:** Pulsing glow border around the preferred mode button in MenuState.

**Technique:** Draw a second `Graphics` roundRect slightly larger than the space panel, using `SPACE_PALETTE.glow` (0x6b8bf5) color, and animate its alpha between ~0.3 and ~0.8 using the ticker in `MenuState.update()`. [VERIFIED: theme.ts:6 for glow color, states.ts:47-56 for drawSpacePanel pattern]

**Why Graphics over filters:** PixiJS GlowFilter requires `@pixi/filter-glow` (extra dependency). A Graphics outline with animated alpha achieves the same visual effect with zero new dependencies and is consistent with the existing `drawSpacePanel` pattern that already uses `SPACE_PALETTE.glow` for border strokes. [ASSUMED]

**Pulse formula:** `alpha = 0.3 + 0.5 * (0.5 + 0.5 * Math.sin(time * 2.5))` gives a smooth oscillation between 0.3 and 0.8 at ~0.4 Hz (one full pulse every 2.5 seconds), which is gentle and non-distracting for children. [ASSUMED]

### WordEntity Interface Update

**What:** The `WordEntity.wordText` field is currently typed as `BitmapText`. It needs updating to support SplitBitmapText (which extends Container, not BitmapText).

**Approach:** Either:
1. Change `wordText` type to `SplitBitmapText` and update all references
2. Add a new `charTexts` field (SplitBitmapText or BitmapText array) alongside `wordText`

Option 1 is cleaner since `wordText` is only used for reading `.text` and setting `.tint` (which SplitBitmapText also has). However, SplitBitmapText does not have `.anchor` -- it uses `pivot` instead. The spawn code sets `newBt.anchor.set(0.5)` which would need adjustment. [VERIFIED: SplitBitmapText source -- uses pivot, not anchor]

**Recommendation:** Use option 1 but store a reference to the SplitBitmapText and set its `pivot` for centering instead of `anchor`. The `AlienContainer.wordLabel` field (typed as `BitmapText | null`) also needs updating.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-character text splitting | Manual char position calculation | `SplitBitmapText` | Handles kerning, font metrics, line layout automatically; each char is a proper BitmapText with tint |
| Glow filter effect | Custom WebGL shader | `Graphics` roundRect with animated alpha | Sufficient for a border glow; no shader complexity needed |
| Text centering for SplitBitmapText | Manual width calculation + offset | `pivot.set(width/2, height/2)` after split | SplitBitmapText's width/height are computed after splitting; set pivot based on bounds |

## Common Pitfalls

### Pitfall 1: SplitBitmapText Anchor vs Pivot
**What goes wrong:** `SplitBitmapText` extends Container, not BitmapText. Container does not have `.anchor`. Using `sbt.anchor.set(0.5)` will crash.
**Why it happens:** The current BitmapText word label uses `anchor.set(0.5)` for centering.
**How to avoid:** Use `sbt.pivot.set(sbt.width / 2, sbt.height / 2)` after the text is set and split. Recalculate pivot when text changes.
**Warning signs:** TypeError at runtime: "Cannot read properties of undefined (reading 'set')"
[VERIFIED: AbstractSplitText extends Container, not BitmapText; Container has pivot but not anchor]

### Pitfall 2: SplitBitmapText Width Before Split
**What goes wrong:** Accessing `sbt.width` before the split completes returns 0, leading to incorrect pivot/centering.
**Why it happens:** With `autoSplit: true` (default), splitting happens in the constructor or on text set, but requires font to be loaded.
**How to avoid:** Set text, then immediately access `.chars` and `.width` -- if autoSplit is true, the split has already occurred by the time the constructor returns (as long as the font is loaded). GameFont is loaded in BootState before any gameplay. [VERIFIED: states.ts BootState loads fonts before transitioning]
**Warning signs:** Word text appears at wrong position (offset from alien center).

### Pitfall 3: Stale Char References After Text Change
**What goes wrong:** Setting `sbt.text = "NEW"` destroys old line/word/char children and creates new ones. Any cached references to old `.chars` entries become invalid.
**Why it happens:** AbstractSplitText.set text() destroys old lines, clears chars/words/lines arrays, then re-splits. [VERIFIED: AbstractSplitText.js:81-89]
**How to avoid:** Always re-read `.chars` after setting `.text`. In `_spawnWord()`, set text once, then reference chars from the SplitBitmapText directly in `_processWordInput()` via `entity.wordText.chars`.
**Warning signs:** Tinting has no visible effect (tinting destroyed objects).

### Pitfall 4: Menu Glow on Null Preferred Mode
**What goes wrong:** Applying glow when `preferredGameMode` is null, violating D-07 (first launch = no highlight).
**Why it happens:** Forgetting the null check.
**How to avoid:** Guard with `if (profile?.preferredGameMode)` before creating the glow graphics. Both buttons start without glow.
**Warning signs:** Glow appears on first launch before any session played.

### Pitfall 5: Word Pool Cleanup with SplitBitmapText
**What goes wrong:** AlienContainer.reset() does not destroy SplitBitmapText children, leading to memory leaks or stale visual state when the pooled item is reused.
**Why it happens:** The current reset() only resets sprite, letterLabel, scale, alpha, visibility. It does not touch wordLabel.
**How to avoid:** When reusing a word alien from the pool, either: (a) destroy the old SplitBitmapText and create a new one, or (b) update `.text` on the existing SplitBitmapText (which triggers a full re-split). Option (b) is more pool-friendly. Reset all char tints to white (0xffffff) when the word text is updated.
**Warning signs:** Gold tint "bleeds" from previous word into new word on the same alien.
[VERIFIED: alien-container.ts:86-96 reset() does not touch wordLabel]

## Code Examples

### Creating SplitBitmapText for Word Label

```typescript
// Source: PixiJS 8.17.1 SplitBitmapText.js + AbstractSplitText.js (verified in node_modules)
import { SplitBitmapText } from 'pixi.js'

// In _spawnWord(), replace BitmapText creation:
if (!ac.wordLabel) {
  const sbt = new SplitBitmapText({
    text: '',
    style: { fontFamily: 'GameFont', fontSize: 38 },
  })
  sbt.y = 2
  ac.addChild(sbt)
  ac.wordLabel = sbt
}
const wordSbt = ac.wordLabel as SplitBitmapText
wordSbt.text = word.toUpperCase()
// Center using pivot (SplitBitmapText has no anchor)
wordSbt.pivot.set(wordSbt.width / 2, wordSbt.height / 2)
```

### Per-Character Tinting on Correct Keypress

```typescript
// Source: Derived from AbstractSplitText.chars (verified BitmapText children with .tint)
// In _processWordInput(), after cursorIndex++:
const GOLD = 0xffd700 // D-01 locked decision
const sbt = activeWord.wordText as SplitBitmapText
for (let i = 0; i < sbt.chars.length; i++) {
  sbt.chars[i].tint = i < activeWord.cursorIndex ? GOLD : 0xffffff
}
```

### Menu Glow Effect

```typescript
// Source: Existing drawSpacePanel pattern (states.ts:47-56) + SPACE_PALETTE.glow
// In MenuState.enter(), after creating mode panels:
const profile = ctx.getActiveProfile()
const preferred = profile?.preferredGameMode ?? null

if (preferred) {
  const targetPanel = preferred === 'letters' ? letterPanel : wordPanel
  const glow = new Graphics()
  const glowPad = 6 // padding outside the panel
  glow.roundRect(
    -letterPanelW / 2 - glowPad,
    -letterPanelH / 2 - glowPad,
    letterPanelW + glowPad * 2,
    letterPanelH + glowPad * 2,
    UI_CONSTANTS.panelCornerRadius + 4,
  )
  glow.stroke({ color: SPACE_PALETTE.glow, width: 3, alpha: 0.6 })
  glow.x = targetPanel.x
  glow.y = targetPanel.y
  this.menuContainer.addChildAt(glow, 0) // Behind the panel
  this.glowGraphic = glow
}

// In MenuState.update():
if (this.glowGraphic) {
  this.glowTime += dt / 1000
  this.glowGraphic.alpha = 0.4 + 0.4 * Math.sin(this.glowTime * 2.5)
}
```

### Fallback: Manual Multi-BitmapText (if SplitBitmapText crashes)

```typescript
// If SplitBitmapText has runtime issues, create individual BitmapText per char:
const chars: BitmapText[] = []
let xOffset = 0
for (const ch of word.toUpperCase()) {
  const bt = new BitmapText({
    text: ch,
    style: { fontFamily: 'GameFont', fontSize: 38 },
  })
  bt.x = xOffset
  bt.tint = 0xffffff
  ac.addChild(bt)
  chars.push(bt)
  xOffset += bt.width
}
// Center the group
const totalWidth = xOffset
for (const bt of chars) {
  bt.x -= totalWidth / 2
}
```

## Open Questions

1. **SplitBitmapText centering accuracy**
   - What we know: SplitBitmapText loses browser kerning, so width may differ slightly from BitmapText
   - What's unclear: Whether the visual centering on the alien will look off
   - Recommendation: Test visually after implementation; adjust pivot if needed. Not a blocking concern.

2. **Pool item reuse with SplitBitmapText**
   - What we know: Setting `.text` on SplitBitmapText destroys and recreates all char children
   - What's unclear: Whether this destroy/recreate cycle per word spawn causes GC pressure at high spawn rates
   - Recommendation: Monitor performance. If GC spikes appear, switch to `autoSplit: false` and manually call `.split()` less frequently.
