## Warnings

### WR-01: Difficulty manager never adapts to wrong keypresses in word mode

**File:** `src/game/states.ts:884-890`

**Issue:** In `_processWordInput`, the `else` branch (wrong key) increments `this.misses` but never calls `this.difficulty.recordResult(false)`. In contrast, `_processLetterInput` calls `this.difficulty.recordResult(false)` on every wrong key (line 839). The difficulty manager therefore remains completely blind to per-character mistakes in word mode: it only adapts when a word escapes off the bottom (the escape handler on line 613 does call `recordResult(false)`). This means word-mode difficulty will not ease back during a bad run the way letter mode does.

**Fix:**
```typescript
} else {
  // Wrong key: dodge animation, no cursor reset
  this.misses++
  this.difficulty.recordResult(false)   // add this line
  if (activeWord.tween === null) {
    activeWord.tween = createDodgeTween()
  }
}
```

### WR-02: Correct per-character keypress does not record a difficulty success

**File:** `src/game/states.ts:854-862`

**Issue:** In `_processWordInput`, the `'correct'` branch (correct key, not yet last character) increments `this.hits` but never calls `this.difficulty.recordResult(true)`. Only `'complete'` (the final character of a word) calls `recordResult(true)`. This means a player who types four out of five letters of every word perfectly but keeps missing the last letter will have the difficulty manager see only misses — it will ease difficulty as if the player is failing, even though they are getting most keystrokes right. The `hits` counter does accumulate correctly for the session results, but the adaptive difficulty signal is misleading.

**Fix:**
```typescript
if (result === 'correct') {
  activeWord.cursorIndex++
  this.hits++
  this.difficulty.recordResult(true)   // add this line
  // Per-character gold tinting ...
```

### WR-03: `letterLabel` is never hidden on word-mode alien containers

**File:** `src/game/states.ts:773-803`

**Issue:** `AlienContainer` always has a `letterLabel` (a `BitmapText`) that is visible by default and reset to `alpha = 1` in `reset()`. When `_spawnWord` acquires a word-pool item and sets `wordSbt.text`, it does not hide `letterLabel`. If the word-pool items share the same `AlienContainer` class (which they do — line 772 casts the pool item directly to `AlienContainer`), the letter label text from a previous letter-mode use (or from construction with an empty string) will be rendered at `y = 2` on top of the word `SplitBitmapText`, which also sits at `y = 2` (line 781). The `letterLabel` is white text with `anchor.set(0.5)` so it will overlay the centre of the word text.

**Fix:** Hide the letter label when spawning a word alien:
```typescript
// After ac.setTexture(...)
ac.letterLabel.visible = false   // suppress single-letter overlay in word mode
```
And restore visibility in `reset()` (or rely on pool contracts — currently `reset()` does not set `letterLabel.visible = true`, which means once hidden it stays hidden across pool reuse, which is actually correct behaviour if word-pool items are never reused as letter-mode items).

## Info

### IN-01: `LevelUpResult.remainingXp` is misleadingly named

**File:** `src/game/states.ts:1052` (also `src/game/progression.ts:14,76`)

**Issue:** `LevelUpResult.remainingXp` stores the cumulative total XP after the session (see `progression.ts:76`: `remainingXp: newTotalXp`). The name "remaining" implies XP left over within a level, but it is actually the full accumulated XP. The computation in `states.ts:1052` that reconstructs the pre-session XP state — `prevTotalXp = this.saveResult.levelUp.remainingXp - this.saveResult.xpGain.totalXp` — works correctly but relies on this non-obvious property. A reader unfamiliar with the data model will misunderstand both the field and the subtraction.

**Fix:** Rename `remainingXp` to `newTotalXp` in `LevelUpResult` and update all read sites. No behaviour change needed.

### IN-02: `SplitBitmapText` pivot set before layout resolves

**File:** `src/game/states.ts:788`

**Issue:** After `wordSbt.text = word.toUpperCase()`, the pivot is immediately set via `wordSbt.pivot.set(wordSbt.width / 2, wordSbt.height / 2)`. `SplitBitmapText` extends `Container`; PixiJS defers layout measurement to the next render tick. Querying `wordSbt.width` synchronously after a text change may return a stale or zero value (the text render pass has not run yet). In the mock, `width` is always `200` so tests pass regardless, but in production the pivot could be set using the previous word's width if the container is being reused from the pool. The `wordWidth` guard on line 798 (`|| 100`) handles the zero case for x-positioning but does not help with the pivot.

**Fix:** Defer the pivot update to after PixiJS has laid out the text, or — since `SplitBitmapText` is centered at the alien's origin — use `sbt.x = -sbt.width / 2` after a forced measurement, or rely on PixiJS `updateTransform`. The safest approach is to call `wordSbt.updateText?.()` if available before reading `width`, or to set `wordSbt.pivot.x` in the `update` tick after at least one frame.

### IN-03: `MockSplitBitmapText.split` mock method is unused

**File:** `tests/__mocks__/pixi.ts:95`

**Issue:** `MockSplitBitmapText` declares a `split: Fn = vi.fn()` method. `SplitBitmapText` in PixiJS v8 does not have a public `split()` method on the instance — `SplitBitmapText` _is_ a split text container. No production code in the reviewed files calls `.split()` on a `SplitBitmapText`. The mock method is dead.

**Fix:** Remove the unused method to keep the mock lean and avoid the impression that `split()` is part of the API contract.

```typescript
// Remove from MockSplitBitmapText:
split: Fn = vi.fn()
```
