## Issues Encountered

### Safeguard test fails on first run — 9 additional orphan keys surfaced (escalated per D-16)

After Task 1's deletions, the new `tests/i18n/keys-used.test.ts` fails with 9 orphan keys that are declared in en.json but have zero literal references in `src/`:

```
progression.xp.earned
progression.avatar.unlocked
levelTitle.cadet
levelTitle.apprenti
levelTitle.pilote
levelTitle.capitaine
levelTitle.commandant
levelTitle.amiral
boot.error
```

Investigation:

- `levelTitle.*` (6 keys): The function `getLevelTitle()` in `src/game/theme.ts:31` returns hardcoded literal strings ('Cadet', 'Apprenti', etc.) from a `LEVEL_TITLES` map. The i18n keys are NEVER consulted. Real tech debt: the rank UI in `states.ts:959` should use `t('levelTitle.cadet')` etc. but it does not.
- `progression.xp.earned`, `progression.avatar.unlocked`, `boot.error`: Searched src/ exhaustively — neither literal nor template-string usage found. These are genuinely orphan in the current codebase.

Per the plan's explicit Task 2 escalation note ("If the test reveals orphans on first run, the executor should still commit the test file as-is, then halt and surface the orphan list to the user before proceeding — see D-16"), and per D-11 (no sweep beyond the audit list), and per D-16 (do not expand the phase mid-flight):

- **Test file committed as-is in `6ee66a7`.**
- **9 orphan keys NOT deleted** in this plan — they are deferred for a follow-up cleanup phase.
- **`pnpm exec vitest run` currently exits non-zero** because of this safeguard. This is the intended behavior per D-12 ("guards against future dead keys regressing in") and the explicit escalation contract in the plan's Task 2 action notes.

**Resolution path (for the next phase / follow-up cleanup):**
- Either (a) wire `getLevelTitle()` and the level-up / xp-earned / boot-error UI through `t()` so the keys are actually used, or (b) delete the 9 orphan keys after explicit user approval. Option (a) is the correct semantic fix; option (b) is a quick mechanical fix.
- Until that follow-up runs, the safeguard test remains red — that is the regression-prevention signal it was designed to provide.
