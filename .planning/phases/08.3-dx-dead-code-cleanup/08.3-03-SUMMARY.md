## Worktree-environment notes

The `prek install --hook-type pre-commit --hook-type commit-msg` command itself was NOT executable inside this worktree because `core.hooksPath` is set (prek refused with "Cowardly refusing to install hooks with `core.hooksPath` set"). However, the destination path that `core.hooksPath` points to (the main clone's `.git/hooks/` directory) ALREADY contains both prek-generated hook files (`pre-commit` and `commit-msg`, both authored by prek with header `# File generated by prek` and matching script ID), and that directory is the effective hooks directory for this worktree as well. Verification therefore exercised the actual installed hooks and proved end-to-end rejection works.

**Follow-up (NOT a blocker — README fix is the deliverable here):** On any future fresh clone where `core.hooksPath` is NOT set, the new README command should run cleanly. If a maintainer ever re-installs hooks on this clone, they will need to either temporarily unset `core.hooksPath` (`git config --unset --local core.hooksPath`) or run `prek install --hook-type pre-commit --hook-type commit-msg` from a clone where the config isn't set. This is per-clone state, not a documentation gap.
