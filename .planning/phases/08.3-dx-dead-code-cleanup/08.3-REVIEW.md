## Warnings

### WR-01: Substring-include matching in i18n safeguard test produces false positives, masking real orphan keys

**File:** `tests/i18n/keys-used.test.ts:41`
**Issue:** The orphan check uses `fileContents.some((c) => c.includes(key))`. When one declared key is a string prefix of another (e.g., `progression.level` is a prefix of `progression.levelUp`), the substring match against the longer key falsely satisfies the shorter key's check. Verified concretely against the current codebase: `progression.level` is declared in `src/shared/i18n/en.json:2` and `fr.json:2` but is never referenced by any `t()` call in `src/`. The only `src/` occurrences of the literal `progression.level` are inside `celebration.ts:41`, which calls `t('progression.levelUp')`. The safeguard test passes despite `progression.level` being a genuine orphan.

The same prefix relationship holds for `profiles.delete` (prefix of `profiles.delete.confirm/.yes/.no`) and `progression.avatar.locked` (prefix of `progression.avatar.locked.message`). In those two cases the shorter keys ARE used elsewhere (`profile-state.ts:199`, `:416`), so today's safeguard is correct by luck. The bug is in the matching strategy, not in the current key set.

This defeats the stated purpose of the test (D-12: "guards against future dead keys regressing in"). A future contributor could add a key like `menu.start` while leaving `menu.startGame` in use, and the orphan would never be caught.

**Fix:** Match each key as a delimited literal rather than a substring. Two dependency-free options:

Option A — quote-anchored regex per key (preferred; handles both `'k'`, `"k"`, and `` `k` ``):
```typescript
function escapeRegex(s: string): string {
  return s.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')
}

// inside the test:
for (const key of enKeys) {
  const pattern = new RegExp(`['"\`]${escapeRegex(key)}['"\`]`)
  const found = fileContents.some((c) => pattern.test(c))
  if (!found) orphans.push(key)
}
```

Option B — bracket the search string with a delimiter that cannot be part of an i18n key, e.g. require the key to be preceded and followed by a non-`[A-Za-z0-9_.]` character:
```typescript
const pattern = new RegExp(`(^|[^A-Za-z0-9_.])${escapeRegex(key)}([^A-Za-z0-9_.]|$)`)
```

Either fix will make the test surface `progression.level` as an orphan, which (a) proves the safeguard works and (b) flags one more genuinely dead key for follow-up cleanup. Note: this finding is itself a candidate for the next quick task — fixing the matcher will likely red the test, requiring the orphan key to be removed in the same change.
